### 1. What Still Holds Perfectly True

* **The eBPF TTL & Checksum Problem (Phase 2.2):** This is your biggest technical bottleneck. When you use Linux `ip_forward` or kernel bridging, the kernel automatically decrements the TTL and updates checksums. Because you are using eBPF `tc` to redirect packets at Layer 2, you bypass the kernel's L3 routing logic. If you don't manually decrement the TTL in eBPF, `traceroute` will show a direct connection (0 hops), which instantly flags you as a MITM.
* **Egress MAC Rewrite (Phase 2.1):** Also mathematically mandatory for stealth. If Victim A sends a packet to Gateway, you intercept it, put it on the wire to the Gateway with *your* MAC address as the source, the Gateway's ARP cache will update to associate Victim A's IP with your MAC. You *must* rewrite the source MAC back to Victim A's MAC before it hits the physical wire to the gateway.
* **ARP Restoration (Phase 5.1):** Still critical. A sudden disconnect that leaves the ARP cache poisoned will cause a Denial of Service for the victim, triggering immediate investigation.

### 2. What Needs Adjustment / Modern Blind Spots

Your plan is excellent, but it misses a few modern detection mechanisms that are now standard in enterprise environments:

#### A. Dynamic ARP Inspection (DAI) & DHCP Snooping

*Your Plan mentions "Switch port security" and "arpwatch", but misses DAI.*
Modern enterprise switches (Cisco, Aruba, Juniper) use DHCP Snooping to build a trusted binding table of MAC↔IP↔Port. If an ARP packet arrives on a port with a MAC/IP mismatch, the switch drops it.

* **Impact:** Your ARP spoof (Phase 1) will silently fail on managed networks.
* **Mitigation:** You can't evade DAI from a rogue host. However, you should add a "DAI detection" mechanism: if you send a poison but don't see the victim's traffic within X seconds, alert the user that the network likely has L2 protections.

#### B. ICMP Time Exceeded Generation (Phase 2.2 Gap)

If you implement TTL decrement in eBPF, you must handle the edge case where `TTL == 1` before decrementing.

* If a packet arrives with `TTL=1` and you just decrement it to `0` and forward it, the next hop will drop it, but *no ICMP Time Exceeded* packet will be generated by that hop because the source MAC is the victim (due to your 2.1 Egress rewrite).
* This breaks `traceroute` entirely, which is a red flag. Your eBPF code (or userspace) needs to generate an ICMP Time Exceeded back to the victim when the TTL hits 0.

#### C. NIC Checksum Offloading (eBPF Implementation Trap)

When modifying the TTL and IP header in Phase 2.2, you will interact with `bpf_l3_csum_replace`.
However, modern NICs use hardware offloading (e.g., `TXCSUM`). The kernel might leave the checksum incomplete (0x0000) expecting the hardware NIC to calculate it. If your eBPF program calculates the full checksum and writes it, the hardware might ignore it, or the hardware might overwrite your correct checksum with a bad one.

* **Fix:** You may need to use `sk_buff->csum` manipulation or force `ethtool -K <iface> tx off` on the interface before attaching your eBPF programs, otherwise your spoofed packets will be dropped by the receiver as invalid checksums.

#### D. eBPF Visibility (New Threat Vector)

Security tools like Falco, Tetragon, and Tracee now monitor the `bpf()` syscall and TC program attachments.

* **Impact:** Even if your network packets are perfect, an EDR (Endpoint Detection and Response) or HIDS will log: `Process "harper" attached TC eBPF program to interface eth0`.
* **Counter:** Add a module to clear `/sys/kernel/debug/tracing/trace_pipe` or avoid using `bpf_trace_printk` in your release builds.

#### E. IPv6 RA Guard

Your Phase 6.4 mentions IPv6/NDP spoofing. Just like DAI for IPv4, modern switches implement RA Guard, which drops Router Advertisements coming from access ports.

* **Mitigation:** You will need to send ICMPv6 Router Advertisement packets with a Hop Limit of 255 to bypass some older RA Guard implementations, but ultimately, L2 IPv6 spoofing is heavily locked down in modern enterprises.
